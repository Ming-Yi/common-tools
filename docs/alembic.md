# Alembic integration

Alembic configuration and revision files belong to each consuming service because that service
owns its models and `AppBase.metadata`.

## Bootstrap a service

Install Alembic in the service's development dependencies and create its async environment:

```bash
alembic init -t async migrations
```

Import every model module explicitly in `migrations/env.py`, then provide the application's
metadata to Alembic:

```python
from alembic import context

from my_service.config import settings
from my_service.models import account, invoice  # noqa: F401
from my_service.models.base import AppBase

config = context.config
config.set_main_option(
    "sqlalchemy.url",
    settings.database_url.replace("%", "%%"),
)
target_metadata = AppBase.metadata
```

Keep the async `run_migrations_online()` produced by Alembic's template. Do not import models by
scanning directories, and never store credentials in `alembic.ini`.

## Development workflow

```bash
alembic revision --autogenerate -m "add invoice status"
alembic upgrade head
alembic check
```

Autogenerated revisions are candidates: review column types, destructive operations, constraint
names, and data transitions before committing them. CI should run `alembic check` in each service
to detect model changes without a revision.

## Deployment workflow

Run `alembic upgrade head` as a dedicated deployment step before starting the new application
version. Do not let every application replica migrate on startup. For database-per-tenant systems,
the consuming service owns the bounded migration orchestration and tenant database discovery.
